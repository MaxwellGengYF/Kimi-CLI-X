# 编写自定义 Skill

Skill（技能）是扩展 Kimi Agent 能力的模块化知识包。通过编写自定义 Skill，你可以将特定领域的工作流、代码脚本、参考文档和模板整合在一起，让 Agent 在处理相关任务时获得精准的上下文指导。

本教程将带你全面了解 Skill 的设计原则、目录结构、编写流程和最佳实践。

## 什么是 Skill

Skill 本质上是一个带有结构化说明的文件夹，Agent 会在遇到特定场景时自动加载其中的内容。与直接在对话中输入冗长提示词相比，Skill 具有以下优势：

- **模块化**：一次编写，多处复用
- **按需加载**：仅在相关场景触发时才进入上下文，节省 Token
- **可携带资源**：可以绑定脚本、文档、模板等实体文件
- **可分发**：可以打包成 `.skill` 文件分享给他人

---

## Skill 的核心设计原则

### 1. 简洁至上

Agent 的上下文窗口是有限的。Skill 的内容会与系统提示词、对话历史和用户请求共享这一空间。因此：

- 只包含 Agent 原本不知道的信息
- 用精简的示例代替大段理论解释
- 尽量将 SKILL.md 控制在 500 行以内

### 2. 自由度匹配

根据任务的"脆弱程度"，为 Skill 中的指导设定合适的约束级别：

| 自由度 | 适用场景 | 推荐形式 |
|--------|----------|----------|
| **高自由度** | 多种合理方案，需根据上下文决策 | 文字描述与原则 |
| **中等自由度** | 有推荐模式，但允许一定变通 | 伪代码 / 参数化脚本 |
| **低自由度** | 对一致性要求极高的易错操作 | 精确可执行的脚本 |

### 3. 渐进式披露

Skill 采用三级加载机制，避免一次性塞入过多信息：

1. **元数据**（`name` + `description`）—— 始终保留在上下文中，用于判断是否触发
2. **SKILL.md 正文** —— 仅在 Skill 被触发后加载
3. **附属资源**（脚本、参考文档等）—— 仅在 Agent 主动读取时加载

因此，核心工作流放在 SKILL.md 中，细节内容拆分到 `references/` 里，并通过明确的指引告诉 Agent 何时去读取。

## Skill 的目录结构

一个标准的 Skill 包如下所示：

```
skill-name/
├── SKILL.md              # 必需：Skill 的核心说明文件
├── scripts/              # 可选：可执行脚本
├── references/           # 可选：按需加载的参考文档
└── assets/               # 可选：模板、图片、字体等输出资源
```

### 各目录说明

- **`SKILL.md`**（必需）
  - 顶部包含 YAML frontmatter，至少包括 `name` 和 `description`
  - `description` 必须说明两点：这个 Skill **是做什么的**，以及 **在什么情况下使用**
  - 正文使用 Markdown 编写操作指令

- **`scripts/`**
  - 放置确定性、可复用的代码
  - 用脚本表达逻辑比用文字更省 Token

- **`references/`**
  - 放置超过 1 万词的大型文档、JSON Schema、API 文档等
  - 在 SKILL.md 中通过相对路径链接，并注明读取时机

- **`assets/`**
  - 放置模板文件、示例图片、字体等辅助输出的资源

**注意**：不要包含 `README.md`、`CHANGELOG.md` 等辅助说明文件，它们会浪费上下文空间。

## 创建 Skill 的完整流程

```
理解需求 → 规划资源 → 初始化目录 → 编写内容 → 测试脚本 → 打包分发 → 迭代优化
```

### 第 1 步：理解需求

收集具体的使用场景和示例。问自己：

- Agent 在什么情况下需要这个 Skill？
- 用户通常会怎么描述相关请求？
- 这个 Skill 需要输出什么格式？质量标准是？

### 第 2 步：规划资源

判断哪些内容适合用文字描述，哪些适合写成脚本，哪些需要链接到外部文档。

### 第 3 步：初始化目录

创建文件夹并建立基本结构。文件夹名称必须与 Skill 名称完全一致。

### 第 4 步：编写内容

先写脚本和参考文档，再提炼核心指令写入 SKILL.md。

### 第 5 步：测试脚本

所有脚本在打包前必须实际运行验证。如果脚本很多且结构相似，至少抽测有代表性的样本。

### 第 6 步：打包

将 Skill 文件夹压缩为 `.skill` 格式的 zip 包。

### 第 7 步：迭代优化

根据实际使用中的反馈，持续精简 SKILL.md、优化触发描述、补充遗漏场景。

## 编写 SKILL.md

SKILL.md 是 Skill 的灵魂。它的质量直接决定了 Agent 能否在正确的时间、用正确的方式使用你的 Skill。

### Frontmatter 格式

文件开头必须包含如下 YAML  frontmatter：

```yaml
---
name: skill-name
description: 简述功能。Use when: (1) 条件A，(2) 条件B...
---
```

**命名规范**：

- 仅使用小写字母、数字和连字符 `-`
- 长度不超过 64 个字符
- 建议用动词开头，如 `gh-address-comments`、`linear-address-issue`

### 正文写作建议

正文应采用**祈使句**，直接告诉 Agent 该做什么。建议包含以下内容：

- **多步骤工作流**：按顺序列出操作步骤
- **决策点**：在需要分支判断的地方给出明确条件
- **输出格式与质量标准**：告诉 Agent 最终输出应该长什么样
- **资源链接**：当需要更多细节时，引导 Agent 读取 `references/` 或运行 `scripts/` 下的文件

例如：

```markdown
## 工作流

1. 读取用户提供的仓库名和 PR 编号
2. 运行 `scripts/fetch_pr.sh <repo> <pr_number>` 获取 PR 详情
3. 若 PR 存在未解决的评论，按文件分组整理评论内容
4. 输出格式：Markdown 表格，列包括文件路径、行号、评论内容、建议修改
```

## 组织附属资源

### scripts/：让代码替你说话

当某个操作步骤非常确定时，写成脚本比用文字描述更高效、更可靠。

示例：

```bash
# scripts/generate-report.sh
python3 -m report_generator --input "$1" --output report.md
```

在 SKILL.md 中引用：

```markdown
运行 `scripts/generate-report.sh <input_file>` 生成报告。
```

### references/：大型文档的归属地

如果你需要引用大段 API 文档、完整配置 Schema 或领域知识库，请将它们放入 `references/`，而不是直接塞进 SKILL.md。

引用方式：

```markdown
关于 API 的完整参数列表，请读取 `references/api-docs.md`。
```

### assets/：输出辅助资源

如果你希望 Agent 在生成内容时套用某个模板（如 PPT 模板、HTML 模板、字体文件），将它们放在 `assets/` 中，并在 SKILL.md 中说明使用方式。

## 测试与打包

### 测试清单

在打包前，请逐项确认：

- [ ] `name` 和 `description` 填写完整且准确
- [ ] 文件夹名称与 `name` 完全一致
- [ ] SKILL.md 中所有脚本路径正确，且脚本已通过实际运行测试
- [ ] 没有多余的文件（如 README.md、CHANGELOG.md、.git 目录等）
- [ ] `references/` 中的大文档有明确的读取触发条件

### 打包命令

进入 Skill 所在的根目录，执行以下命令：

```bash
cd <skills-root>
zip -r my-skill.skill my-skill
```

打包后得到的 `my-skill.skill` 就是一个标准的 Skill 分发包，可以直接安装使用。

## 安装与使用

Skill 可以安装在多个位置，Agent 会自动扫描并加载：

| 级别 | 路径 |
|------|------|
| 用户级 | `~/.config/agents/skills/`、`~/.kimi/skills/`、`~/.claude/skills/` |
| 项目级 | `.agents/skills/` |
| 自定义 | 通过 `--skills-dir` 参数指定 |

将 Skill 文件夹或 `.skill` 压缩包放入上述任一目录即可生效。

## 最佳实践与示例

### ✅ 推荐做法

1. **先想触发条件，再写内容**
   `description` 决定了 Agent 会不会用这个 Skill。写得越清晰具体，误触发和漏触发就越少。

2. **一个 Skill 只做一件事**
   不要把多个不相关的工作流硬塞到一个 Skill 里。小而专的 Skill 更容易维护和组合。

3. **用相对路径引用资源**
   所有内部链接都应相对于 SKILL.md，确保 Skill 被移动到不同目录后仍能正常使用。

4. **避免过深的引用嵌套**
   `references/` 里不要再套多层子目录，保持扁平结构，降低 Agent 的读取成本。

### ❌ 常见错误

- **描述过于笼统**：`description: 帮助写代码` —— 这种描述几乎会在任何对话中触发，造成干扰。
- **正文过于冗长**：把整本规范书都复制进 SKILL.md，导致上下文被挤爆。
- **缺少决策点**：只给了最终结果要求，没告诉 Agent 中间步骤该怎么做。
- **脚本未测试**：打包的脚本在目标环境中无法运行，导致 Agent 执行失败。

### 完整示例

假设我们要创建一个帮助整理 GitHub PR 评论的 Skill：

**目录结构：**

```
gh-address-comments/
├── SKILL.md
└── scripts/
    └── fetch_comments.py
```

**SKILL.md：**

```markdown
---
name: gh-address-comments
description: 整理 GitHub PR 中待处理的评论。Use when: (1) 用户提到要回复或处理 PR 评论，(2) 需要汇总代码审查意见。
---

# gh-address-comments

## 工作流

1. 从用户输入中提取仓库全名（如 `owner/repo`）和 PR 编号
2. 运行 `scripts/fetch_comments.py <repo> <pr_number>` 获取评论数据
3. 过滤出状态为 `open` 且非 bot 发布的评论
4. 按被评论的文件路径分组
5. 对每条评论，给出：文件路径、行号、评论原文、建议的代码修改
6. 输出为 Markdown 表格

## 输出标准

- 表格必须包含 `文件`、`行号`、`评论`、`建议修改` 四列
- 如果评论没有明确建议，在最后一列标注 "需进一步讨论"
- 只输出表格，不要额外总结
```

## 总结

编写一个好的 Skill，关键在于**精准的触发描述**、**精简的核心指令**和**可靠的配套资源**。牢记以下三点：

1. `description` 必须说清楚"做什么"和"何时用"
2. SKILL.md 控制在 500 行以内，细节拆到 `references/`
3. 脚本必须经过测试，打包前清理冗余文件

现在，你可以开始编写自己的第一个 Skill 了！
